# AgentFab Glossary

**Fabric** — A set of AI agents that collaborate to complete tasks. Contains one Conductor, one or more specialist agents, a shared volume, and a definition file. Also referred to as an "agentic fabric."

**Definition File** (`agents.yaml`) — Single source of truth for system topology. Lists all agents with their names, models, capabilities, and endpoints. Versioned. Agents reload on change without restart.

**Conductor** — The orchestrating agent. Sole user-facing interface. Decomposes requests into task graphs, spawns agents, handles escalations.

**Agent** — A specialized AI entity with its own LLM, prompt, tools, and persistent knowledge. Defined in YAML. Runs as a goroutine (local), container (Docker), or pod (Kubernetes).

**Agent Definition** — YAML schema specifying an agent: `name`, `purpose`, `capabilities`, `model` (`provider/model-id` format), `tools`, `escalation_target`, `verify`, `review_guidelines`, and other configuration. See [architecture.md](architecture.md#agents).

**Default Profile** — Built-in agent definition for common roles (conductor, architect, designer, developer). Enables zero-config setup via `agentfab init`. All are overridable.

**Task Graph** — DAG generated by the Conductor. Nodes are tasks (each assigned to an agent). Edges are dependencies. See [architecture.md](architecture.md#task-graphs).

**Orchestration Loop** — Bounded FSM for iterative work between agents (e.g., implement → review → revise). Hard `max_transitions` cap prevents infinite loops. See [architecture.md](architecture.md#orchestration-loops).

**LoopContext** — A `DataPart` carrying FSM routing metadata through agent-to-agent messages. Contains the loop definition, current state, and task context. Excluded from LLM input.

**Knowledge Base** — Per-agent persistent store. Contains a knowledge graph (`graph.json`), node documents (Markdown), and cold storage. Nodes carry provenance metadata (confidence, source, TTL, hits). See [architecture.md](architecture.md#knowledge-system).

**Cold Storage** — Archive for infrequently-accessed knowledge nodes. Nodes with few hits and no recent access are moved here. Decision-tagged nodes are exempt. Expired nodes are permanently purged after a retention period (default 3 years).

**Curation** — LLM-based consolidation of a knowledge graph. Triggered when node count exceeds a threshold (default 50). Merges overlapping nodes, preserves decisions and high-confidence entries. Removed nodes go to cold storage.

**Decision Injection** — Decision-tagged knowledge nodes are scored by keyword relevance against the current task or request and injected only when above a relevance threshold. This ensures project-wide constraints propagate to related tasks without flooding unrelated ones.

**Supersession** — A `supersedes` edge linking a new decision to the old one it replaces. Superseded nodes are excluded from lookups.

**Special Knowledge File** — Markdown file loaded into an agent's context on every invocation. Contains domain conventions or standards. Read-only from the agent's perspective.

**Message** — Unit of inter-agent communication. Contains `id`, `request_id`, `from`, `to`, `type`, and multipart `parts`. Types: `task_assignment`, `task_result`, `escalation`, `review_request`, `review_response`, `status_update`.

**Part** — Content unit within a Message. Three types: `TextPart` (text), `FilePart` (URI to shared volume), `DataPart` (structured JSON).

**Shared Volume** — Storage accessible to all agents. Contains the definition file, interaction logs, and agent artifacts.

**Scratch Space** — Ephemeral per-agent storage for script execution. Destroyed after use. The only tier that permits deletes.

**Escalation Chain** — Path for resolving uncertainty. Each agent escalates to its `escalation_target`. The Conductor escalates to the user.

**Circuit Breaker** — Stops a task or loop on budget breach, consecutive errors, nonsensical output, or timeout. Triggers escalation.

**Verification Gate** — Optional `verify` block in an agent's definition. Runs a shell command after the tool loop ends. On failure, errors are injected back and the agent gets extra iterations.

**Task Scope** — Controls iteration depth: `small` (no review loop), `standard` (max 3 transitions), `large` (max 5). Set by the Conductor during decomposition.

**Sandbox** — Isolated execution environment for agent tools. Stripped environment, OS-level filesystem restrictions, timeout enforcement. See [architecture.md](architecture.md#sandboxing).

**Runtime** — Abstraction layer between agent logic and deployment mode. Five interfaces: `Communicator`, `Storage`, `Discovery`, `Lifecycle`, `Meter`. Implementations: local, Docker, Kubernetes.

**Eino** (`github.com/cloudwego/eino`) — The Go LLM library used by AgentFab. Provider adapters (Anthropic, OpenAI, Google, OpenAI-compatible) come from `eino-ext`.

**Agent Manifest** (`agents/manifest.json`) — SHA-256 checksums of all agent definition files. Generated by `agentfab init` or `agentfab agent compile`, verified on `agentfab run`.

**Agent Compilation** — Process of generating structured YAML agent definitions from plain Markdown descriptions. Run via `agentfab agent compile`. Each `.md` file becomes an agent; a conductor is auto-added if absent.

**Response Envelope** — JSON metadata block in agent LLM output. Carries verdict, summary, artifacts, and issues. Envelope-first parsing with legacy fallback.

**Decomposition Template** — YAML file providing a starting-point task graph pattern. The LLM adapts templates rather than generating graphs from scratch.

**Disambiguation** — LLM-based clarity check on user requests. If ambiguous, the Conductor asks a clarifying question before decomposition.

**Agent Chat** — Direct interaction with a specific agent via the `!` command in the REPL. Bypasses the Conductor's decomposition for quick questions, small tasks, or knowledge queries. Chat responses are parsed for amendment and escalation markers.

**Local Mode** — Default execution mode. All agents run as goroutines within a single process, communicating via in-memory Go channels. No network I/O or serialization overhead.

**Distributed Mode** — Execution mode where each agent runs as a standalone gRPC server (`agentfab agent serve`). Agents communicate over protobuf with mTLS. The conductor generates a CA and per-agent certificates at startup.

**Budget** — Token and cost limits enforced per agent and per task. Defined as `max_input_tokens`, `max_output_tokens`, `max_total_tokens`, and `max_cost_usd` in the agent definition. Checked before every LLM call; on breach, the agent stops and sends a failed result.
